# Sim-U-Kraft 新版重写当前进度与后续计划

本文是后续进度与计划的统一维护文件。接口调用规范见 [接口用法.md](接口用法.md)。

## 项目定位

新版 Sim-U-Kraft 以旧版玩法体验为规则来源，但底层按 NeoForge 1.21.1、LDLib2、SQLite 和服务层架构重新实现。

核心原则：

- 服务端是权威端，客户端只负责显示和请求。
- 长期数据进入 SQLite，运行时缓存必须有作用域和失效策略。
- 旧版规则优先还原，旧版低性能实现不直接继承。
- NPC、建筑、城市、经济、材料和寻路都通过 Service 串联，避免 GUI 或 Packet 写业务。
- 任何新功能都要考虑多人协作、存档安全、服务器性能和后续扩展。

## 当前已完成

### 基础工程

- NeoForge 1.21.1 工程可编译运行。
- Java 21、Gradle、资源处理链路可用。
- LDLib2 已接入，当前依赖为 CurseMaven 版本。
- 主类、注册表、方块、实体、声音、网络包基础结构已形成。
- SQLite 主数据库和建筑结构数据库已接入。

### 城市系统

- 城市核心方块可作为管理入口。
- 支持创建城市、查询城市、成员权限、城市资金、财政流水。
- 支持城市区块购买、相邻校验、资金扣除、失败回滚。
- 客户端城市地图、区块边界、城市核心标记和基础交互已可用。
- 城市 POI 已作为住宅、岗位、建筑生态的基础数据来源。

### 建筑系统

- 建筑盒、建筑资源目录、建筑列表和建筑预览已可用。
- 支持建筑旋转、预览移动、边界渲染和空间占用判断。
- 支持建造任务、建造进度、材料消耗、完成登记。
- 建筑完成后可登记已放置建筑、住宅床位 POI、完成音效和弹窗消息。
- 住宅控制盒可以显示建筑、居民、容量、床位和边界。
- 建筑拆除链路已具备基础清理能力。

### 材料系统

- 已有通用材料策略、材料请求、材料结果、材料缓存和通用容器访问。
- 建造师可从紧贴工作方块的箱子读取材料。
- 支持基础模式、专家模式、可替代材料组和跳过列表。
- 缺材料提示已改为右侧滑出弹窗，支持合并重复消息。
- 材料弹窗支持物品图标和长文本自适应高度。

### NPC 系统

- 市民实体、持久化数据、姓名、性别、年龄、寿命、健康、饥饿、技能等字段已具备。
- 支持城市绑定、住宅绑定、工作点绑定、职业状态和雇佣/解雇。
- NPC 死亡后保留数据并登记死亡，不再直接删除记录。
- 新 NPC 加入住宅时优先生成在床边安全位置。
- 支持夜间回家、第二天返回工作岗位、无任务建筑师回岗位待命。
- 支持职业统一手持物和动作表现，建筑师有任务时右手持圆石并挥动。

### 寻路系统

- 已有 `CitizenNavigationService` 统一入口。
- 支持近距离真实寻路、远距离按配置传送兜底。
- 支持行走、奔跑、跳跃、游泳、梯子/藤蔓/脚手架攀爬。
- 主线程采集快照，异步只处理不可变快照，不直接访问世界。
- 支持路径缓存、快照缓存、限流、失败冷却和卡住传送。
- 支持木门开门；不主动开铁门、活板门、栅栏门。
- 支持 Alt+P 客户端调试渲染和寻路测试/压力测试指令。
- 已加入拥堵协调和局部让行逻辑，仍需继续打磨拐角和身位校准。

### UI 与提示

- 主要 GUI 使用 LDLib2。
- 城市核心和 NPC 信息界面已改为普通窗口框架，不再继承 LDLib2 `Editor`，避免触发 UI Scale 自动设置。
- 已新增 `SimuKraftWindowFrame` 作为普通窗口容器。
- 已新增右侧滑出弹窗，用于系统消息、材料提醒、金钱提示和建筑完成提示。
- 弹窗支持重复消息合并、Logo 显示、物品图标和高度自适应。

### 农田系统

- 农田盒（`nsuk_farmland_box`）已可作为农场控制入口，右键打开 LDLib2 普通窗口。
- 配置进 SQLite（`farmland_boxes` 表）+ SavedData 兜底，含作物、作业区域、运行状态。
- 复用既有系统：放置时自动登记 `FARMLAND` POI 并限制在城内，破坏时注销；农民经既有雇佣链以 `workplacePos=箱子位置` 绑定。
- `FarmlandFarmingService` 实现真种真收：移动农民→耕地浇水→消耗箱内真实种子种植→等原版自然生长→收获成熟作物真实掉落物入箱（满则掉地），绝不凭空产出。
- 支持小麦、胡萝卜、马铃薯、甜菜根（满铺）和西瓜、南瓜（藤蔓棋盘）。
- 设置区域为 `FarmlandAreaScreen`（键盘操作的自由视角界面，实现 `FreeCameraScreen` 标记接口）：打开即进入自由视角可飞行观察，不暗化世界、实时线框预览（复用 `BuildingBoundsRenderer`）；面向相对模型（长=纵深/宽=横向/离盒距离/左右平移），默认以玩家朝向、农田盒外一格为近边底线，不再以盒子为中心；确认/取消后退出自由视角、回到玩家视角，自定义 min/max 发服务端校验（限尺寸、防重叠）。键鼠 Mixin 的自由视角白名单已改为按 `FreeCameraScreen` 接口判断。
- 区域界面 ESC/取消会丢弃未保存选区（不发设置包并清除候选线框）；自由视角飞行 R 可把区域对齐到当前相机朝向。
- 视线对准农田盒持续超过 1 秒，`FarmlandHoverPreview` 会请求并显示该盒已保存作业区域的线框（轻量请求/响应包，多人安全，仅展示）。
- 切换作物为弹出菜单 `FarmlandCropScreen`：列出全部作物并标注当前选中，点选后发服务端设置。
- 仓储箱不再手动绑定，开始耕作及每个工作周期由 `FarmlandBoxService.resolveAdjacentChest` 自动检测紧贴农田盒六个面的容器，其它位置不考虑；无相邻箱时不能开始/暂停作业。
- 性能：只处理运行中的盒子，每盒独立节流，游标分片，不全量扫描区域，未加载区块跳过。

### 经济与 HUD

- 城市资金、财政流水、HUD 同步已具备。
- 住宅租金服务已有基础 tick 结算。
- HUD 显示城市、资金、人口、日期等基础信息。

## 当前主要风险

- NPC 在窄口、墙角、半格碰撞体附近仍可能不按理论路径校准身位。
- 寻路调试线正确不代表原版 `PathNavigation` 一定严格按线行走，需要继续限制局部目标和转向时机。
- 材料系统必须持续避免每 tick 扫箱子，后续新增职业工作也要复用缓存。
- 城市核心、NPC 信息等 LDLib 窗口需要继续实测是否完全不会改动玩家 UI Scale。
- 建筑拆除、床位、居民绑定、工作点绑定之间还要加强一致性修复。
- 数据库 schema 后续扩展前要规划版本字段和迁移策略。

## 已知代码问题清单（扫描结果，待修）

一次代码扫描发现的具体缺陷，按严重度排列。✅=已对照源码确认，⚠️=待复核。

高危：

- ✅ 雇佣越权：`NpcHireAssignPacket` / `NpcHireListRequestPacket` 只校验 16 格距离和是否可雇，不校验玩家对该城市/工作点的权限，任意玩家可雇佣、调度、传送别人城市的 NPC。
- ✅ 缓存清理大小写不匹配：`BuilderConstructionService.clearServerCaches`(:185) 用未小写的 `serverKey` 匹配，而运行时 key 在 `:551` 被 `toLowerCase`。Windows 盘符大写导致关服/切档时 `LEVEL_RUNTIMES` 清不掉，同会话换存档串档、内存增长（其它服务大小写一致，仅此处不一致）。

中危：

- ✅ 全量实体扫描：`CitizenTeleportService.findCitizenEntity`(:74) 每次查实体都 `getAllEntities()` 全量遍历，且在查找方法里顺手 discard 重复实体（副作用）；被寻路/雇佣高频调用，O(实体数)/次。
- ✅ 全量实体扫描：`CitizenNavigationService` 的 `countLoadedCitizens`(:413)/`findNearest`(:436) 在 `requestMove` 准入控制里再次 `getAllEntities()` 全量扫描。
- ✅ 建城无回滚：`CityCoreCreateCityPacket:81-86`，`createCity` 落库后若 `assignInitialArea` 失败直接 return 不回滚，玩家得到无领地城市且被 `findPlayerCity` 非空永久锁定无法再建城。
- ✅ 拆除丢物：`PlacedBuildingDemolitionService:30` 用 `setBlock(AIR, 3)` 不 `dropResources`，建筑内箱子/熔炉等内容物全部丢失。
- ✅ 批量存档 no-op：`CitizenManager.saveToSqlite`(:53-57) 方法体为空、`save()` 返回空 tag，市民完全依赖增量写；崩溃时未增量落库的字段会丢（需复核增量是否覆盖全部字段）。
- ⚠️ 收租占用判定不一致：`ResidentialRentService` 按整建筑计费，但占用用 `homeId→POI` 与 `getPoiAt` 两套判定；多床建筑只要 1 床有人就收整建筑租金，POI 替换后可能漏收。
- ⚠️ 人口增长限流失效：`PopulationGrowthService.tick:22-28`，`maxPerInterval` 只限生成、不限 `fillVacantHomes` 全量重分配，每间隔对每城市全量扫描市民。
- ⚠️ 入住空转：`CitizenHousingService.spawnCitizensForVacantHomes`，`ensureCitizen` 返回 null 时不 break，可能反复刷实体；依赖 `setHome` 即时反映到 `vacantHomes` 查询。

低危：

- ⚠️ `CityPoiManager.replacePoi:217`/`releaseCity:165`：`posIndex.remove` 用非 immutable 的 `pos()`，插入用 `.immutable()`，极端情况下残留 pos→id 映射。
- ⚠️ `CitizenManager.getOrCreate:138`：`aiQueue.contains()` 对 `ConcurrentLinkedQueue` 是 O(n) 线性扫描，且可能重复入队。
- ⚠️ `ResidentialRentService:159`：同一玩家待发收入提示用 `put` 覆盖，时间跳跃时早一条被静默丢弃（仅显示偏差）。

## 后续计划

### P0：稳定当前基础闭环

目标：先让现有城市、NPC、建造、材料、寻路在高频使用下不出脏数据、不掉刻、不堵死。

任务：

- 回归验证城市核心和 NPC 信息窗口不会强制 UI Scale 自动。
- 修复 NPC 在墙角、两方块边缘、铁栅栏、完整方块前提前转向导致的蹭墙问题。
- 给寻路执行层增加更严格的 waypoint 到达判定、身位校准和局部目标锁定。
- 验证死亡 NPC 在雇佣列表、世界人口、城市人口中的显示规则一致。
- 验证材料弹窗同时显示“需要：原材料”和可替代材料，长文本不截断。
- 给建造完成、缺材料、租金等消息做重复合并和冷却检查。
- 整理压力测试指令输出，让“不动、失败、传送、正在排队”的 NPC 状态可见。

验收：

- 64 个活动寻路 NPC 不持续重算，不出现大面积反向拥堵。
- 200 个移动请求只按配置限流入队，服务器 TPS 稳定。
- 缺材料不会刷消息，不会刷物品，不会拖慢建造。
- 死亡 NPC 不可雇佣，数据仍可追溯。

### P1：完善建造与材料闭环

目标：从选择建筑到雇佣、寻路、要材料、施工、完成、登记 POI、拆除清理形成稳定闭环。

任务：

- 建造师没有任务时保持在工作岗位附近待命，不抢任务、不乱跑。
- 建造任务恢复时校验建筑盒、建筑师、材料缓存和目标建筑记录。
- 建筑完成后统一发送音效、弹窗、HUD/界面刷新。
- 住宅床位 POI 与建筑记录双向一致，破坏床、恢复床、拆除建筑都能正确更新。
- 材料配置增加更多旧版兼容组，保持基础模式和专家模式差异清晰。
- 建造速度继续按 NPC 等级、配置和休息状态计算。

验收：

- 断线/重进/重启后建造任务能恢复或安全中断。
- 建筑完成后床位容量和住宅控制盒立即正确。
- 拆除建筑不遗留床位 POI、边界渲染或居民错误绑定。

### P2：完善 NPC 日常行为

目标：让市民在住宅、工作点、闲逛、死亡、休息之间稳定切换。

任务：

- 完善白天上班、夜晚回家、无任务待命、空闲闲逛的统一调度。
- 所有“到达目标点”的功能都走寻路优先、传送兜底。
- 继续完善木门开门、拥堵让行、卡住一分钟传送。
- 为农民、商业员工、工业员工、物流员工、守卫预留工作行为接口。
- 恢复饥饿下降、食物获取、吃东西和工作效率影响。

验收：

- NPC 不会因目标区块未加载而强制加载新区块。
- NPC 不会在不可达目标前永久顶墙。
- 同一 NPC 不会同时持有多个移动请求。

### P3：城市管理界面与权限

目标：把城市核心变成城市管理主入口，不再让建造盒承担过多管理职能。

任务：

- 城市核心新增完整雇佣管理页：空闲 NPC、已雇佣 NPC、岗位容量、解雇。
- 城市核心新增建筑/住宅/岗位概览页。
- 城市财政页显示租金、工资、区块购买、建筑完成等分类流水。
- 成员权限页支持更完整的权限修改和操作反馈。
- 城市升级页定义人口、建筑、资金、声望等条件。

验收：

- 所有城市管理操作服务端校验权限。
- 客户端界面只展示服务端响应，不自行推断最终状态。

### P4：经济、生产与物流

目标：让城市从“能建造”进入“能经营”。

任务：

- 每日结算服务：租金、税收、工资、维护费。
- 农田生产：种植、收获、库存。
- 商业建筑：营业收入、顾客需求、利润。
- 工业建筑：原料、产出、生产周期。
- 仓储与物流：库存、物流点、物流员工任务。
- 城市满意度、声望、安全值对经济产生影响。

验收：

- 经济数值全部进入财政流水。
- 生产链不会每 tick 全量扫描库存或建筑。
- 物流系统不能刷物品，失败必须有可解释状态。

### P5：测试、性能与发布整理

目标：降低多人协作风险，保证长期维护成本可控。

任务：

- 建立 GameTest 或命令式测试场景：寻路、建造、材料、床位、死亡、租金。
- 增加压力测试基准：100/200/500 NPC，不同寻路配置下的 TPS 和排队状态。
- 清理过长文件，超过 500 行的服务优先拆分职责。
- 为新增 SQLite 表补齐索引、版本字段和迁移计划。
- 整理语言文件 key，避免同义 key 分散。
- 发布前冻结方块 ID、物品 ID、实体 ID、主要网络包语义和数据库核心表结构。

验收：

- `compileJava` 稳定通过。
- 压测指令可复现，输出可读。
- 关键功能有明确回归清单。

## 协作约定

- 新增服务入口或稳定调用链时，同步更新 [接口用法.md](接口用法.md)。
- 完成一个大模块后，同步更新本文的“当前已完成”和“后续计划”。
- 不要把临时方案写成新的散乱 md；除非确实需要独立设计文档，否则优先维护本文。
- 修改 common 业务逻辑后检查是否需要同步客户端显示、网络包、语言文件和配置界面。
- 所有能改变世界或数据的操作必须先有服务端校验，再有持久化，再有客户端同步。
