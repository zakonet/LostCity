# Sim-U-Kraft 新版重写计划书

## 1. 项目定位

新版 Sim-U-Kraft 不是对旧版代码的简单迁移，也不是逐文件照搬旧工程。新版目标是保留旧版的核心玩法和玩家体验，重新设计底层架构，清理历史遗留技术债、冗余代码和低性能实现，形成一个适合 Minecraft 1.21.1 与后续版本长期维护的城市模拟模组。

旧版项目已经积累了大量功能，但同时存在以下问题：

- 功能分散在大量工具类和界面类中，职责边界不清晰。
- 网络包、GUI、业务逻辑、存档读写互相耦合。
- 许多系统直接操作 SavedData，缺少统一服务层。
- 同类功能重复实现，例如商业、工业、农田、物流雇佣数据各自维护。
- 旧 GUI 数量庞大，部分界面逻辑与服务端业务混杂。
- NPC、建筑、物流等模块存在性能瓶颈和历史兼容残留。
- 部分缓存没有严格按世界、维度或服务端隔离。
- 配置、语言文件、资源路径和旧测试之间缺少统一约束。

新版的核心思路是：

> 以旧版功能为需求来源，以新版架构重新实现，不直接继承旧版技术债。

同时，新版城市设计可以参考 Minecraft 原版村庄机制进行优化。原版村庄的优点不是界面复杂，而是底层规则清晰：床决定居住容量，工作站决定职业，POI 决定寻路目标，集会点组织村民活动，声望和安全事件影响村庄状态。新版 Sim-U-Kraft 可以吸收这些设计思想，但不要直接照搬原版内部实现。城市核心只作为市长和官员连接城市管理系统的终端，不作为城市内部 POI 或城市实体组成部分。

## 2. 重写目标

### 2.1 功能目标

新版最终应覆盖旧版主要玩法：

- 城市核心与城市管理。
- 城市资金、财政、升级、人口和权限。
- 城市区块认领、购买、地图可视化。
- NPC 市民、家庭、职业、雇佣与工作调度。
- 建筑模板、建筑箱、控制盒、材料统计和施工任务。
- 商业、工业、农田、物流和仓储系统。
- HUD、Toast、地图、城市核心 UI、配置 UI、指南书。
- 必要的第三方 Mod 集成。

### 2.2 架构目标

新版应满足：

- 模块职责清晰。
- 业务逻辑优先放在服务层，不写在 GUI 或网络包里。
- SQLite 作为长期持久化层，SavedData 只作为生命周期桥接和兜底。
- 客户端缓存只保存展示数据，不作为权威数据来源。
- 服务端永远是权威方，所有资金、认领、建筑、雇佣、物品操作必须服务端验证。
- 网络包只负责传输和调度，不直接塞满业务逻辑。
- 数据结构要适合高频查询和大量 NPC 场景。
- 持久化数据统一进入 SQLite，不再新增散落 JSON、SK 或临时文件。
- 高频变更使用增量保存，定时全量同步只作为兜底。
- 每个大模块都能独立测试和逐步替换。

### 2.3 性能目标

新版必须重点解决旧版性能问题：

- 避免每 tick 全量扫描所有 NPC、建筑、物流节点。
- 避免 O(n³) 或无界遍历。
- 高频查询使用索引缓存，例如 UUID、城市 ID、区块 long key。
- 地图渲染使用 Region 纹理缓存，不能每帧大量 CPU 填色。
- NPC AI 使用分片调度，不允许所有 NPC 同 tick 重任务。
- 物流和仓储必须有脏标记和增量刷新。
- 客户端 GUI 不得频繁向服务端请求全量数据。
- 所有长期缓存必须有世界、维度、玩家或城市作用域。

## 3. 核心设计原则

### 3.1 服务层优先

每个大系统都应有明确服务入口：

| 系统 | 服务层建议 |
|---|---|
| 城市 | `CityService` |
| 区块认领 | `CityClaimService` |
| 经济 | `EconomyService` |
| 财政流水 | `FinanceLedgerService` |
| NPC | `CitizenService` |
| 职业 | `JobService` |
| 雇佣 | `EmploymentService` |
| 建筑 | `BuildingService` |
| 施工 | `ConstructionService` |
| 规划 | `PlanningService` |
| 商业 | `CommercialService` |
| 工业 | `IndustrialService` |
| 农田 | `FarmlandService` |
| 物流 | `LogisticsService` |
| HUD 同步 | `HudSyncService` |
| 地图缓存 | `SimuMapManager` / `CityMapOverlayService` |

网络包、GUI、事件只调用服务层，不直接修改底层数据。

### 3.2 数据层与业务层分离

SQLite 负责：

- 按关系表持久化城市、NPC、区块、POI、财政流水等长期数据。
- 为城市 ID、NPC UUID、区块 long key、POI ID 等高频查询字段建立索引。
- 支持单城市、单 NPC、单 POI、单区块的增量写入。
- 在服务器停止和低频兜底同步时接收全量快照。

SavedData 负责：

- 维持 Minecraft 世界生命周期中的 Manager 实例。
- 保存运行时内存对象和索引。
- 作为现有 `CityManager.get(level)`、`CitizenManager.get(level)` 等入口的桥接。
- 必要时标记 dirty，保留 Minecraft 原生保存机制的兜底能力。

SavedData 不负责：

- 权限判断。
- 资金扣款规则。
- 建筑是否合法。
- NPC 是否能工作。
- 物流物品能否取出。
- 直接承担长期数据主存储。

这些规则应放到服务层。

当前已经确认的 SQLite 表包括：

```text
cities
city_members
finance_transactions
city_chunks
city_pois
citizens
citizen_skills
```

后续新增系统时，优先新增专用表和索引，例如：

```text
employment_records
construction_tasks
building_templates
logistics_nodes
warehouse_items
city_reputation
city_safety
```

`SimuSqliteStorage` 只作为对外门面，SQL 细节按模块拆分为仓储类，例如：

```text
CitySqliteRepository
CitizenSqliteRepository
CityPoiSqliteRepository
CityChunkSqliteRepository
FinanceSqliteRepository
```

### 3.3 客户端只展示

客户端可以拥有：

- HUD 缓存。
- 地图纹理缓存。
- 城市区块 overlay 缓存。
- GUI 展示数据。
- 建筑预览数据。

客户端不能决定：

- 资金变化。
- 区块归属。
- 物品增减。
- NPC 雇佣结果。
- 建筑任务是否创建。
- 城市权限是否通过。

### 3.4 兼容但不继承屎山

旧版功能需要保留，但旧版实现不必保留。

允许直接复用的内容：

- 资源文件。
- 建筑模板。
- 语言键。
- 注册 ID。
- 玩家可见的交互流程。
- 性能较好的独立模块，例如旧版 SimuMap Region 缓存思路。

不建议直接复用的内容：

- 大型 GUI 代码。
- 巨型 NetworkManager。
- 混合业务逻辑的工具类。
- 重复的雇佣数据实现。
- 直接在 packet 中改存档的逻辑。
- 无作用域的全局缓存。

### 3.5 参考原版村庄机制优化城市设计

新版城市不是简单的“一个城市核心 + 一堆菜单”。城市核心的定位是城市管理终端，它负责让市长和官员访问城市、财政、地图、成员、升级等系统；真正的城市生态应该像原版村庄一样，由一组可检测、可索引、可调度的城市要素组成。

可以参考的原版机制：

| 原版村庄机制 | 新版城市对应设计 | 说明 |
|---|---|---|
| POI | 城市兴趣点 `CityPoi` | 住宅、工作站、仓库、集会点、农田、商店都可以注册成 POI；城市核心不属于 POI |
| 床 | 居住容量 | 住宅建筑提供床位/容量，决定城市人口上限和 NPC 可入住数量 |
| 工作站 | 职业岗位 | 控制盒、农田箱、商店、工厂、物流箱提供岗位，玩家通过雇佣把 NPC 绑定到岗位，不自动分配 |
| 集会点 | 城市广场/中心点 | NPC 空闲时可以去集会点，而不是所有 NPC 都围绕城市核心 |
| 职业绑定 | 雇佣系统 | NPC 只有被市长或官员雇佣后才获得职业，岗位失效时职业或工作状态需要重新计算 |
| 村庄边界 | 城市区块范围 | 城市只在已认领区块内生效，建筑和工作站必须归属某个城市 |
| 声望 | 城市声望/满意度 | 玩家行为、税率、工资、住房和安全可以影响城市满意度 |
| 袭击/安全 | 城市安全系统 | 后续可扩展怪物袭击、防御建筑、守卫职业 |
| 繁殖条件 | 人口增长条件 | 住房、食物、工作、满意度达标后城市自然增长人口 |

新版城市建议拆为以下数据层：

| 数据 | 作用 |
|---|---|
| `CityData` | 城市基础信息：名称、UUID、核心位置、等级、资金、成员 |
| `CityChunkData` | 城市区块归属和边界 |
| `CityPoiData` | 城市 POI 索引：住宅、工作站、仓库、集会点等 |
| `CityPopulationData` | 人口、住房容量、出生/迁入统计 |
| `CityJobData` | 岗位、职业需求、空缺、雇佣状态 |
| `CitySafetyData` | 安全、威胁、守卫、防御设施 |
| `CityReputationData` | 声望、满意度、玩家关系 |

城市运行时不要每 tick 扫描方块寻找建筑，而应该采用“注册 + 失效”的方式：

1. 放置城市功能方块时注册 POI。
2. 破坏方块或区块失效时注销 POI。
3. 建筑完成时批量注册住宅容量、岗位、仓库入口等 POI。
4. NPC AI 查询 `CityPoiIndex`，而不是临时遍历世界。
5. 城市每日结算使用缓存数据，只有脏标记时重算。

这样既能保留原版村庄“由床、工作站、POI 组成生态”的优点，又能避免旧版那种到处扫描、到处保存、到处直接改数据的性能问题。

## 5. 模块重写计划表

### P0：基础工程与注册

| 编号 | 模块 | 重写目标 | 关键要求 | 验收标准 |
|---|---|---|---|---|
| P0-1 | 主类 | 建立稳定 NeoForge 1.21.1 主入口 | 注册、事件、网络分离 | 游戏能启动，服务端不加载客户端类 |
| P0-2 | 注册系统 | 重建方块、物品、实体、菜单注册 | ID 与旧版兼容，注册类职责单一 | 注册内容能进入游戏 |
| P0-3 | 资源整理 | 整理模型、贴图、语言、建筑资源 | 不复制无用残留资源 | 缺失模型和语言 key 降到最低 |
| P0-4 | 网络骨架 | 按功能拆分 packet | 禁止巨型 NetworkManager | 每类 packet 可单独维护 |
| P0-5 | 配置骨架 | 建立服务端/客户端配置 | 配置项有默认值和语言 | 可读取配置并应用 |

### P1：城市核心系统

| 编号 | 模块 | 重写目标 | 关键要求 | 验收标准 |
|---|---|---|---|---|
| P1-1 | 城市数据 | 重写城市数据与 SQLite 持久化 | 城市 UUID、成员、资金、等级、核心位置，关系表与索引 | 创建后重进世界不丢数据 |
| P1-2 | 城市服务 | 新建 `CityService` | 创建、删除、重命名、查询统一入口 | GUI 和 packet 不直接改数据 |
| P1-3 | 权限系统 | 重写市长/官员/市民权限 | 权限判断集中化 | 所有管理操作经过权限验证 |
| P1-4 | 成员管理 | 添加/移除/设置权限 | 使用 UUID 为主，名字只做展示 | 玩家改名后不丢权限 |
| P1-5 | 区块认领 | 重写城市区块归属 | long chunk key 索引 | 查询区块归属 O(1) |
| P1-6 | 区块购买 | 资金扣款 + 区块认领 | 服务端验证相邻、归属、权限 | 无法越权买区块 |
| P1-7 | 城市升级 | 条件、费用、等级效果 | 配置驱动 | 升级结果可保存 |
| P1-8 | 城市 POI 索引 | 参考原版 POI 重写城市要素索引 | 功能方块注册/注销，建筑完成批量注册 | 城市住宅、岗位、仓库、集会点可 O(1)/低成本查询 |
| P1-9 | 居住容量 | 住宅决定人口上限 | 类似原版床机制，但使用建筑容量 | 城市人口不能超过有效住房容量 |
| P1-10 | 城市集会点 | NPC 空闲行为目标 | 不强制围绕城市核心 | NPC 可前往广场/中心点活动 |
| P1-11 | 城市声望和满意度 | 为后续人口增长和税收服务 | 可由住房、工作、税率、安全影响 | 城市状态页可显示满意度 |

### P2：经济与财政

| 编号 | 模块 | 重写目标 | 关键要求 | 验收标准 |
|---|---|---|---|---|
| P2-1 | 经济服务 | `EconomyService` 统一资金入口 | 金额两位小数，拒绝非法数值 | 所有扣款入账都走服务层 |
| P2-2 | 财政流水 | 新增交易记录 | 类型、金额、原因、操作者、时间 | 财政界面可查看最近记录 |
| P2-3 | 资金权限 | 官员/市长可操作资金 | 市民无权限 | 越权 packet 无效 |
| P2-4 | 收入支出 | 接入建筑、商业、工业、物流 | 交易原因明确 | HUD 和财政页同步刷新 |
| P2-5 | 经济配置 | 价格、税率、维护费配置 | 可调默认值 | 改配置后规则生效 |

### P3：HUD、Toast 与同步

| 编号 | 模块 | 重写目标 | 关键要求 | 验收标准 |
|---|---|---|---|---|
| P3-1 | HUD 同步 | `HudSyncService` 统一同步 | 服务端权威，每秒去重同步 | HUD 不显示旧数据 |
| P3-2 | HUD 渲染 | 保留旧版顶部格式 | 本地化、缓存宽度、异常保护 | 显示城市、资金、星期、人口 |
| P3-3 | HUD 配置 | 位置、开关、缩放 | 客户端配置独立 | 玩家可调整 HUD 位置 |
| P3-4 | Toast | 重写事件提示系统 | 队列、过期、去重 | 城市/建筑/经济事件可弹提示 |

### P4：地图系统

| 编号 | 模块 | 重写目标 | 关键要求 | 验收标准 |
|---|---|---|---|---|
| P4-1 | 地图主体 | 保留旧版 Region 缓存思想 | 512x512 region、后台扫描、纹理上传 | 地图显示真实地形 |
| P4-2 | 地图作用域 | 世界 + 维度隔离 | 单人/多人缓存分离 | 不同世界不串图 |
| P4-3 | 城市 overlay | 区块归属叠加层 | 当前城市、其他城市、核心标记 | 地图上归属清晰 |
| P4-4 | 交互 | 缩放、拖动、悬浮框 | 不滚出画布，裁切正确 | 操作流畅 |
| P4-5 | 性能清理 | 过期 region 释放 | 避免 NativeImage 泄漏 | 长时间打开不卡顿 |

### P5：NPC 与市民系统

| 编号 | 模块 | 重写目标 | 关键要求 | 验收标准 |
|---|---|---|---|---|
| P5-1 | NPC 数据 | UUID 索引、城市索引、职业索引 | ConcurrentHashMap，分片 tick | 大量 NPC 查询稳定 |
| P5-2 | NPC 持久化 | SQLite 关系表 + 增量保存 | 数据版本号，按 NPC UUID 更新，技能独立表 | 重进世界数据不丢，单个 NPC 变化不重写全部 NPC |
| P5-3 | NPC 实体 | 市民实体与数据解耦 | 实体只是运行时表现 | 离线数据不依赖实体存在 |
| P5-4 | 名字/性别/皮肤 | 资源驱动 | 随机但可持久化 | NPC 展示一致 |
| P5-5 | 家庭系统 | 家庭、配偶、子女关系 | 防循环引用 | 家庭关系可查询 |
| P5-6 | 住宅绑定 | NPC 与住宅建筑绑定 | 住宅容量限制，参考原版床位归属思想 | 城市人口与住宅一致 |
| P5-7 | AI 调度 | 分片 tick + 状态机 | 不全量重任务 | 多 NPC 不明显卡顿 |
| P5-8 | POI 寻路目标 | NPC 从 `CityPoiIndex` 获取住宅、岗位、集会点 | 不扫描世界，不临时找方块 | NPC 能稳定找到工作站和住所 |
| P5-9 | 人口自然增长 | 参考原版繁殖条件重写 | 住房、食物、工作、满意度达标才增长 | 人口增长可控且可配置 |

### P6：职业与雇佣

| 编号 | 模块 | 重写目标 | 关键要求 | 验收标准 |
|---|---|---|---|---|

## 6. 技术债清理清单

| 旧问题 | 新版处理方式 |
|---|---|
| 巨型 NetworkManager | 按模块拆分 packet 注册和处理 |
| GUI 内写业务逻辑 | GUI 只发请求，业务进服务层 |
| 多套雇佣数据重复 | 统一 EmploymentService 和槽位模型 |
| SavedData 承担业务规则 | SavedData 只持久化，规则进 Service |
| 全局静态缓存串世界 | 所有缓存加 worldId + dimensionId 作用域 |
| NPC 全量 tick | 分片 tick、状态机、任务队列 |
| 地图每帧 CPU 画大量格子 | Region NativeImage + GPU texture |
| 物流每次全量扫描 | 容器脏标记 + 物品索引缓存 |
| 直接信任客户端 packet | 服务端统一验证权限、距离、维度、资金、物品 |
| 配置和语言残留多 | 每个模块迁移时同步清理无用 key |
| 旧 Mixin 盲目继承 | 逐个复核，能不用就不用 |

## 7. 数据兼容策略

当前项目尚未正式发布，因此新版当前阶段优先使用干净的 SQLite 关系表结构，不再为了未发布阶段的临时 NBT 或通用 SNBT 单表结构做复杂兼容。

### 当前阶段数据原则

- 长期数据统一进入 SQLite。
- 不再新增散落 JSON、SK 或临时文件作为主存储。
- Manager 可以保留 SavedData 生命周期桥接，但 SQLite 是实际持久化层。
- 表结构变化时可以直接迁移开发存档或重建测试存档。
- 在正式发布前应冻结核心表结构，并补充 `dataVersion` 迁移机制。

### 正式发布前需要稳定的内容

- 方块 ID。
- 物品 ID。
- 实体 ID。
- 城市 UUID。
- NPC UUID。
- 建筑模板资源路径。
- 语言 key 中玩家可见的主要文本。
- 关键 SavedData 名称或提供迁移读取器。

### 正式发布前可以继续重构的内容

- Java 包名。
- 内部类结构。
- 服务层接口。
- 网络包结构。
- GUI 实现。
- 缓存结构。
- 任务调度逻辑。

### 数据版本建议

正式发布前，所有 SQLite 主表建议加入或配套记录：

```text
dataVersion
createdAt
lastUpdatedAt
```

读取旧数据时：

1. 判断是否存在旧字段。
2. 转换为新版数据结构。
3. 标记 dirty。
4. 下次保存写新版格式。

## 8. 性能验收指标

| 场景 | 目标 |
|---|---|
| 100 个 NPC | 服务端 tick 无明显卡顿 |

| 多人同时打开 GUI | 不重复全量广播无关数据 |
| 快速点击物品操作 | 服务端防重复、防刷物品 |
| 切换世界/维度 | 缓存不串数据 |

## 9. 开发执行规则

1. 每个模块先写数据模型，再写服务层，再写网络包，最后写 GUI。
2. 不允许 GUI 直接改 SavedData。
3. 不允许 packet 直接执行复杂业务逻辑。
4. 不允许客户端决定最终结果。
5. 所有服务端操作必须验证玩家、维度、距离、权限、资金和目标状态。
6. 每个高频系统必须有缓存失效策略。
7. 每个跨 tick 系统必须有异常隔离，单个任务失败不能拖垮全局。
8. 每个新增配置必须同步语言文件和配置界面。
9. 每个新增网络包必须说明方向和用途。
10. 每次完成模块必须运行编译检查。

## 11. 最终验收标准

新版重写完成后，应满足：

- 玩家能从城市核心创建城市。
- 城市能管理成员、权限、资金、区块、升级。
- HUD 能显示城市状态并实时同步。
- 地图能显示真实地形和城市区块。
- NPC 能生成、保存、分配职业、执行工作。
- 建筑能选择、预览、统计材料、创建施工任务、由 NPC 建造。
- 商业、工业、农田能形成生产和消费循环。
- 物流系统能安全管理物品网络。
- 配置系统可用，指南书可用。
- 旧版核心资源和存档数据有迁移方案。
- 服务端长期运行没有明显内存泄漏或 tick 卡顿。
- 不依赖第三方 Mod 时仍能正常运行。

## 12. 总结

新版 Sim-U-Kraft 的方向不是“把旧代码搬进新项目”，而是“用旧版功能作为设计需求，重新实现一个干净、可维护、高性能的城市模拟框架”。

重写的重点不是一次性追平旧版所有功能，而是建立正确的架构顺序：

```text
数据模型 -> 服务层 -> 网络协议 -> 客户端缓存 -> GUI 展示
```

只要这个顺序稳定，后续建筑、NPC、生产、物流等复杂系统都可以逐步接入，而不会再次形成旧版那种难以维护的技术债。
